# Aperture v1 — Engineering Case Study

## Context

**Aperture** is a token-efficiency layer for Composio-powered LLM agents. It
sits as a Streamable-HTTP MCP proxy between the LLM client and Composio's
Tool Router, intercepts the six meta tools, and ships three compounding
optimizations: a cross-agent execution cache (Component A), hot-path token
attribution observability (Component B), and an offline schema-description
optimizer with a Haiku/Sonnet judge (Component C).

This case study covers what I learned building it across four deep-fix
iterations on `v1-fixes`. I'm calling out three specific bugs that only
surfaced under live API testing, the engineering judgments they forced,
and the gaps that I am intentionally leaving named (not hidden) for
follow-up.

---

## Three bugs mocks would never have caught

### 1. The silent-pass false-accept

The LLM judge runs each prompt through Haiku with the original schema and
again with the candidate schema, then compares `tool_use.name` and the
normalized arguments. When the equality check passed, the candidate was
accepted.

I wired the live run. Anthropic returned 400 errors on every call (see bug
#2). The error path returned `JudgeOutcome(tool_name=None, tool_input=None)`.
Both `original` and `candidate` were `None`. **`None == None` is `True`.**
Every prompt "agreed." Every candidate was "accepted." The budget tracker
showed `$0.00 spent` and `0 calls` — the only signal that anything was
wrong.

The fix: in `live` mode, if both outcomes are empty (no `tool_use` AND no
text), record a hard failure with reason `anthropic_call_failed` rather
than letting `None == None` silently pass. A test fakes the empty-outcome
case and asserts the failure reason.

> **Lesson.** When a result looks too clean — an integration test that
> passes with $0.00 spent — the success path may be a no-op. Audit call
> counts and budget telemetry, not just return values.

### 2. The schema-shape 400 cascade

Composio's `client.tools.get()` returns OpenAI-shape schemas with a
`parameters` key. Anthropic's `client.messages.create(tools=...)` requires
`input_schema`. Every Anthropic call from this codebase was 400-erroring
with `"tools.0.custom.input_schema: Field required"`.

Earlier sessions had reported "live LLM judge accepted GITHUB_CREATE_ISSUE
rewrite for $0.0194" — almost certainly that 400 cascade plus the silent-pass.
The fix was a `_to_anthropic_tool()` adapter that detects either shape and
normalizes. After the fix, a real run made 522 successful calls
(241 Haiku + 26 Sonnet) and produced two judge-accepted candidates, both
of which the safety filter correctly blocked because they were write tools.

> **Lesson.** Every external integration needs one live round-trip before
> "wired" is a defensible claim. Mocks reproduce shapes you specified;
> they don't expose contracts you got wrong.

### 3. The stale artifact / correct code trap

A reviewer flagged that `_overlay.json` shipped 18 tools — 2 of which
`policy.yaml` classified as `write` operations. I checked the code. The
`_overlay_safe` filter rejected `operation_type in {write, auth}`. The
test passed. The code was right.

The artifact was wrong. It had been generated by an earlier code path
that didn't have the filter. Regenerating it with the current code path
(which I had to actually run) produced 0 entries — the truthful state.

> **Lesson.** When you add a gate, every artifact downstream of that gate
> is suspect until regenerated. The test suite verifies the function; it
> doesn't verify the file the function produced six commits ago.

## Engineering decisions worth defending

**Defense-in-depth, not single-point gates.** The schema overlay is gated
by four independent checks: the rewriter never strips safety keywords;
`_overlay_safe` requires `operation_type == "read"` AND
`validation_cases_run >= 50`; the proxy's `SchemaOverlay.reload` applies
the same read-only positive list at startup (so a hand-edited overlay
can't apply rewrites to mutating tools); the LLM judge silent-pass guard
prevents false-accepts on transport errors. The 522-call live run
verified the chain: judge accepted 2 write tools, the next layer down
blocked both.

**Positive lists, not deny lists.** I tightened `_overlay_safe` from
"reject write/auth" to "accept only read." `unknown` operation types are
blocked by default. This means a tool that hasn't been classified yet
can never accidentally land in the overlay just because policy.yaml
hasn't been updated.

**Quality-graded artifacts with explicit metadata.** Without Anthropic
credits, I can't ship an `llm_judged` overlay. Instead of pretending or
shipping nothing, I added a `quality_level` field with values
`llm_judged` and `structural_only`. The latter relaxes the case-count
threshold but stamps the file with a `warning` string and the dashboard
surfaces a yellow banner. Operators can choose their risk level with
full disclosure.

**No truncation fallback.** The rewriter used to fall back to
"truncate to 18 words" when no rules matched. That risked dropping
safety/auth keywords from descriptions. I removed it; the rewriter now
returns `[]` and the upper layer records `rewriter_no_candidate` cleanly.

## What "done" actually means

I scoped this carefully. The **code** is well-engineered: separated
concerns, type-checked, defensively guarded, adversarially reviewed,
234 tests, 0 ruff findings. The **product** has named gaps: the overlay
ships at `structural_only` quality (4 read tools, 9 fields, 64 tokens
saved) because Anthropic credits ran out mid-validation; the original
plan's "100 real-session baseline" was never collected; benchmark
end-to-end runs producing actual `aperture_full` token-savings numbers
haven't been committed.

What I *didn't* do: claim the overlay is production-grade when it isn't.
The README labels it `structural_only`, the dashboard surfaces a warning
banner, and the file's own metadata says `quality_level: structural_only`.
A future engineer who funds Anthropic and runs the LLM-judged path gets
a one-line recipe in both the docs and the commit message.

## What this project taught me about working with AI agents

The hardest part of this project wasn't writing the code — it was
catching the four times an AI agent (Codex or a subagent) reported
"done, all wired" when key pieces were not actually wired. The
silent-pass bug, the schema-shape bug, and the stale-artifact bug all
hid behind subagent reports of success. The discipline that mattered
was running fresh verification commands myself, reading the actual
output, and pushing back when claims didn't match evidence.

That discipline — **evidence over claims** — is what I'd bring to any
team. AI agents accelerate the building. Verifying what they actually
built is what separates a codebase that looks good from one that holds
up under load.
